# singbox-manager 路线图

## 当前定位

项目已经不再是早期 `v0.1.x beta` 阶段，而是：

- 可发行
- 可自用/小规模使用
- 以 CLI + TUI + systemd 运维闭环为核心
- 不做 Web 面板

目标保持不变：

- 单机 / 小规模优先
- 围绕 sing-box 配置、用户、流量、订阅、部署诊断展开
- 保持二进制轻量、依赖少、部署简单

## 近期优先级

### P1

1. 发布流程更稳
   - 版本号 / `Cargo.lock` / `CHANGELOG` / tag 一致性检查
   - 发版前自检脚本
   - 减少“本地能发，CI 因锁文件或版本漂移失败”

2. 诊断能力继续增强
   - `sb doctor` 持续补常见故障项
   - 覆盖证书、端口冲突、服务状态、订阅对外可达性
   - 错误信息尽量直接告诉用户下一步该怎么修

3. 配置/订阅兼容性继续补齐
   - 各协议生成逻辑回归测试
   - 更多 sing-box 配置边界情况
   - 继续减少“格式对，但客户端不认”的问题

4. 运维侧稳定性
   - backup/restore 再做边界校验
   - 安装脚本/升级脚本继续减少环境假设
   - 降低对发行版差异和路径差异的敏感度

### P2

1. 更完整的 CLI 自检/修复建议
2. 更细的日志与错误分级
3. 更多自动化测试覆盖
4. 进一步清理历史兼容逻辑和文档欠账

## 明确不做

- 不做 Web 面板
- 不做复杂 RBAC / 多租户
- 不做平台化后端
- 不做“花哨但收益低”的 TUI 鼠标交互

## 判断标准

后续版本优先看三件事：

1. 部署后是否更稳
2. 出问题时是否更容易定位
3. 发版和升级时是否更少踩坑
